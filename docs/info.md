## How it works

# Technical Specification: Quad-Digit Hexadecimal Display System

This system functions as a **hexadecimal visualization peripheral** controlled via a **Serial Peripheral Interface (SPI)** operating in slave mode. It is designed to process and display 16 bits of data, representing four hexadecimal characters.

---

### Data Transmission Protocol
To ensure data integrity and proper synchronization, the following protocol must be observed:

*   **Transaction Structure:** The system expects a single, continuous SPI transaction. The **Chip Select (CS)** line must remain asserted (Low) for the entire duration of the data transfer.
*   **Data Payload:** The primary payload consists of **16 bits** of information, corresponding to the four hexadecimal digits to be displayed.
*   **Extended Configuration:** To modify system parameters, a **third byte** may be appended to the transaction.

### Refresh-Rate Configuration
The display refresh-rate is user-configurable and is determined by the value of the third transmitted byte. 

*   **Input Range:** The refresh-rate parameter ($RR$) accepts an integer value between **0 and 15**.
*   **Computational Logic:** The internal refresh frequency ($f_{refresh}$) is derived from the system clock ($clk$) using a power-of-two prescaler:

$$f_{refresh} = \frac{clk}{2^{RR}}$$

## How to test

### 1. Hardware Interfacing
Establish a connection between the device and a **SPI Master interface**. Ensure the following signals are correctly mapped:

*   **SCLK (Serial Clock):** Generated by the Master.
*   **MOSI (Master Out Slave In):** For data transmission to the display.
*   **MISO (Master Out Slave In):** For reading the GPIO input port.
*   **CS/SS (Chip Select):** Active in low.


---

### 2. Refresh-Rate Optimization
The refresh frequency should be set within the **kilohertz (kHz)** range to ensure a flicker-free visual output. When defining the Refresh-Rate ($RR$) parameter, account for your specific system clock speed ($clk$).

For example, if your system clock is $10 \text{ MHz}$:
*   Setting $RR = 12$ results in: $10,000,000 / 2^{12} \approx 2.44 \text{ kHz}$.
*   Setting $RR = 13$ results in: $10,000,000 / 2^{13} \approx 1.22 \text{ kHz}$.

---

### 3. Initialization and Data Transmission
The communication flow is divided into an initialization phase and subsequent operational updates:

#### Initial Configuration
During the **first transaction**, transmit a total of **three bytes**:
1.  **Bytes 1 & 2:** The 16-bit hexadecimal data payload.
2.  **Byte 3:** The chosen $RR$ configuration value ($0–15$).

> **Note:** The **CS** (Chip Select) line must remain **Low** for the entire 24-bit sequence to ensure the configuration is registered correctly.

#### Subsequent Updates
For all following transactions, you may opt to send only the **two data bytes** (16 bits). The system will retain the previously configured refresh-rate until it is either power-cycled or explicitly updated via a 3-byte transaction.
